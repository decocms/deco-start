Confirmed gaps (ranked by impact)

1. service.version is never set from CF_VERSION_METADATA
instrumentWorker() reads decoRuntimeVersion option (defaults to hardcoded "5.0.0") and stamps it as deco.runtime.version
It does NOT read env.CF_VERSION_METADATA?.id to set the OTel-standard service.version
Why this matters: dashboards correlate errors / latency with specific deployments via service.version. Without it, “did the regression land in deployment X?” is unanswerable
Fix: in bootObservability(), read env.CF_VERSION_METADATA?.id and stamp it as service.version in spanAttributeFloor AND in the logger floor. Files: src/sdk/otel.ts:168-214
2. Per-span attribute floor only covers framework-created spans, not CF auto-instrumented ones
spanAttributeFloor (deco.runtime.version, deployment.environment) is applied inside the configureTracer bridge in otel.ts:137-149 — only spans created through withTracing() get it
CF runtime auto-instrumented spans (fetch/KV/R2/DO) come out without these attributes
Why this matters: ~90% of spans in a typical request come from CF auto-instrumentation. Without the floor, dashboards can’t filter those by environment or runtime version
Fix options:
(a) Move stamping into ingest Worker: if a resource attribute is missing, fill from the request’s known service. Requires the ingest Worker to know per-service defaults — leaks coupling.
(b) Accept the gap. Filter dashboards by service.name only (resource attribute, always present). deco.runtime.version becomes “best-effort for framework spans”
(c) Use CF Tail Workers to enrich spans post-hoc (heavy)
Recommendation: (b) for now, document explicitly. Revisit if a real dashboard need surfaces.
3. Sampling: head-only via CF Destinations, augmented by a tail worker (RESOLVED)
CF Destinations only supports head_sampling_rate — decided at trace start, before status is known. The original goal of "100% errors + 1% baseline" is now met by a three-channel model, see docs/observability.md → "Error capture — three-channel model". Summary:
  - CF Destinations carries 1% sampled traces + 1% sampled info/warn logs (cheap bulk volume).
  - Direct-POST in-worker carries 100% of framework logger.error(...) and 100% of OTel metrics (CF Destinations can't carry either).
  - Cloudflare Tail Worker (deco-otel-tail, Strategy B) carries 100% of uncaught throws, exceededCpu/Memory outcomes, and raw console.error from third-party SDKs.
The tail worker lives in decocms/stats-lake/ingestion/otel-tail. Producer-side wiring is one line in wrangler.jsonc: `tail_consumers: [{ service: "deco-otel-tail" }]`. See MIGRATION_TOOLING_PLAN.md D-8 for the decision record.
4. Logger output is JSON-in-stringValue, not native OTel log attributes
defaultLoggerAdapter does console.log(JSON.stringify({ level, msg, timestamp, ...attrs }))
CF Destinations wraps that into an OTLP LogRecord with body.stringValue = "<the JSON>". The attrs object isn’t exposed as native log attributes
Our ingest Worker maps body.stringValue → otel_logs.Body verbatim. Queries need JSONExtract(Body, 'level', 'String') to filter by level
Why this matters: ClickStack UI auto-detects severity from body.stringValue substring matching, so “INFO” / “ERROR” show up correctly (confirmed empirically). But filtering by custom attributes (requestId, userId) requires JSON extraction at query time
Fix options:
(a) Have the logger emit severityText and severityNumber to a separate sidecar that’s understood by CF Destinations. Looks hard — CF only sees console output
(b) Accept the JSON-in-body shape, document the JSONExtract pattern for queries
(c) Change the logger to log key-value pairs as separate console.log calls, but that breaks correlation
Recommendation: (b). Document with example queries in @decocms/start observability README.

Brief: instrument @decocms/apps for end-to-end observability
Context

The deco platform pipes telemetry from site Workers → CF Destinations → deco-otel-ingest (https://deco-otel-ingest.deco-cx.workers.dev) → stats-lake ClickHouse → ClickStack UI (hyperdx.clickhouse.cloud). The framework (@decocms/start) emits service identity (service.name, service.version from CF_VERSION_METADATA), a root deco.http.request span, per-section spans, cache decision spans, admin protocol spans, and W3C traceparent-injection helper injectTraceContext(headers) exported from @decocms/start/sdk/observability.

What’s missing is per-app commerce instrumentation. VTEX, Shopify, Nuvemshop, etc. all issue many outbound fetches per request — search, catalog, OMS, intelligent-search, checkout. Today these are CF-auto-instrumented as flat fetch spans with no semantic grouping. Operators can’t answer “is intelligent-search slow today?” without grep-style URL filtering.

Investigation tasks

Audit createInstrumentedFetch — find its current implementation in this repo (likely under vtex/utils/ or utils/). For each:
Does it use withTracing from @decocms/start? If yes, what’s the span name shape?
Are spans named per-endpoint (e.g. vtex.intelligent-search.product_search) or just per-client (vtex.fetch)?
Are HTTP-level attributes set (http.method, http.url, http.status_code, http.response.body_size)?
Does it set STATUS_CODE_ERROR on 5xx responses or on thrown errors?
Is response timing recorded as a histogram metric (e.g. vtex_request_duration_ms with endpoint label)?
W3C trace propagation — does createInstrumentedFetch inject traceparent headers? If not, import injectTraceContext from @decocms/start/sdk/observability and call it before each outbound fetch. Confirms downstream correlation if upstream is OTel-aware.

// utils/tracedFetch.ts (new file)
import { withTracing, injectTraceContext, getMeter, setSpanAttribute } from "@decocms/start/sdk/observability";

export interface TracedFetchOptions {
  provider: string;            // e.g. "vtex"
  operation: string;           // e.g. "intelligent-search.product_search"
  attributes?: Record<string, string | number | boolean>;
}

export async function tracedFetch(
  input: Request | URL | string,
  init: RequestInit | undefined,
  opts: TracedFetchOptions,
  underlyingFetch: typeof fetch = fetch,
): Promise<Response> {
  const headers = new Headers(init?.headers);
  injectTraceContext(headers);
  const url = typeof input === "string" ? input : input instanceof URL ? input.toString() : input.url;
  const method = init?.method ?? "GET";
  const start = performance.now();

  return withTracing(
    `${opts.provider}.${opts.operation}`,
    async () => {
      setSpanAttribute("http.method", method);
      setSpanAttribute("http.url", redactUrl(url));   // strip query secrets like authToken
      for (const [k, v] of Object.entries(opts.attributes ?? {})) setSpanAttribute(k, v);

      let res: Response;
      try {
        res = await underlyingFetch(input, { ...init, headers });
      } catch (err) {
        recordDuration(opts, performance.now() - start, "error");
        throw err;
      }

      setSpanAttribute("http.status_code", res.status);
      const len = res.headers.get("content-length");
      if (len) setSpanAttribute("http.response.body_size", Number(len));
      recordDuration(opts, performance.now() - start, String(res.status));
      return res;
    },
  );
}

function recordDuration(opts: TracedFetchOptions, ms: number, status: string) {
  getMeter()?.histogramRecord?.("commerce_request_duration_ms", ms, {
    provider: opts.provider,
    operation: opts.operation,
    status,
  });
}
Notes:

Error status on 5xx and on thrown errors is handled by withTracing itself (PR 179 sets SpanStatusCode.ERROR when the wrapped function rethrows). For 5xx responses we don’t throw — we just rely on the http.status_code attribute for filtering, which is the OTel-recommended approach (status codes ≠ span errors unless the client treats them as errors).
We rely on getMeter()?.histogramRecord?.(...) from PR 179. Single generic histogram commerce_request_duration_ms with provider + operation labels is more useful for cross-provider dashboards than per-provider names; matches how http_request_duration_ms is shaped upstream.
redactUrl strips known-sensitive query params (authToken, cookie, vtex_session, etc.) consistent with the existing sanitization in vtex/utils/fetch.ts.

# Fourth Context: Observability Architectural Refinement (PRs A/B/C/D — May 22 2026)

After PR 179 landed and rolled out to a few storefronts, three structural gaps surfaced in real traffic:

1. **Fragmented observability.** Three telemetry channels (CF Destinations, direct-POST in-worker, tail worker) produced overlapping but uncorrelated views. Same request appeared in three places without a stable join key, so operators couldn't follow a request end-to-end.
2. **Context gaps.** Logs and tail records lacked stable `request.id` / `trace.id`, so logs and traces couldn't be joined. Tail rows had no method, URL, colo, or runtime cost attributes. The "false-positive ERROR flood" (`outcome=canceled` rows that aren't actually errors) was burying real errors at ~3K/hour.
3. **Cost-effectiveness.** 100% direct-POST traces from the framework would scale linearly with traffic. We need sampled bulk + 100% errors + 100% metrics — and the sampling decision needs to align with what we already do at CF Destinations.

## Phase 4 JSON-body lift bug (post-merge fix, May 22)

After PR A merged and `deco-otel-ingest` redeployed, the JSON-body lift in `logsToRows` was suppressed for 100% of production traffic. Original guard:

```ts
if (Object.keys(baseAttrs).length === 0) {
  // ...attempt JSON.parse and lift fields into LogAttributes
}
```

Intent: "don't overwrite attributes from direct-POST logs that already have structured LogAttributes". But Cloudflare Workers Observability auto-injects two attributes (`name`, `cloudflare.invocation.sequence.number`) into *every* LogRecord — even from a bare `console.log("hello")`. So `baseAttrs.length === 0` was never true in production, and Phase 4 became a no-op.

Fixed in `fix/otel-ingest-phase4-merge-attrs` (PR #7 in stats-lake):
- Removed the `length === 0` guard.
- Attempt JSON parse on any `body.stringValue` shaped like JSON.
- Merge with `{ ...redact(promoted.attrs), ...baseAttrs }` — existing attributes win on key collision, so direct-POST structured logs are still preserved.

Lesson worth carrying forward: a "skip if non-empty" guard against a platform-injected map is the kind of bug only real production traffic surfaces. Unit tests for `logsToRows` would have caught it — see P3 below.

2. `baggagio-tanstack` — medium traffic.
3. `miess-01-tanstack` — medium traffic.
4. `casaevideo-storefront` — highest traffic, biggest customer. Goes last.

**Shippable now:** open identical-shape PRs per storefront — one-line `@decocms/start` bump, no code changes, CF Workers Builds handles deploy. ~10 minutes per storefront if `wrangler.jsonc` is already wired.

**Architecturally refined:** sites missing the `tail_consumers` / `observability` block need that wired first. `deco-audit-observability` already flags them. The refined version is a single batched audit-and-fix PR per storefront, not one PR per finding.

**Risk:** the framework changes are additive, no API removed, but rolling out a behavior change to every storefront in a 24-hour window risks one shared regression across the fleet. **Mitigation:** soak each storefront 24h before promoting the next.

**Evolution path:** once 4/4 storefronts have soaked clean, the audit script flips from warn to block (P4) — making any future storefront missing the bindings break CI rather than silently miss telemetry.

## P3 — Vitest coverage on stats-lake/ingestion/otel-ingest

Flagged at fix-time of the Phase 4 bug. The merge-attrs logic in `logsToRows` is exactly the kind of code where a hidden guard silently no-ops.

**Shippable now:** five test cases covering the main behaviors:
- JSON-shaped body with no pre-existing LogAttributes → fields lifted.
- JSON-shaped body with CF-injected attributes only → fields lifted *and* CF attrs preserved.
- JSON-shaped body with key collision against pre-existing attribute → pre-existing wins.
- Non-JSON body → body left as-is, no lift attempted.
- Malformed JSON body → body left as-is, no throw.

~80 LOC, contained to one test file. No tradeoff.

**Architecturally refined:** snapshot-test the full OTLP→ClickHouse-row transform with a golden fixture of recorded real OTLP payloads. Catches regressions across the whole `logsToRows`/`tracesToRows`/`metricsToRows` surface, not just the merge-attrs bug. ~2x the work.

**Evolution path:** ship the five-case version first to close the immediate gap; pull a real OTLP payload into a golden fixture next time a similar bug bites.

## P4 — Promote `deco-audit-observability` from warn → block (per D-16)

**Shippable now:** flip every clean repo to `--mode block` in one batched PR. Fast but breaks CI in any repo that hasn't been fixed yet.

**Architecturally refined:** per-repo, per-section gradual promotion:
1. Land the audit script in CI as warn-only on every `decocms/*` and `deco-sites/*` repo.
2. Wait one cycle (a week) to see which findings hit which repos.
3. For each clean repo, flip to `--mode block` via a separate PR.
4. For dirty repos, fix findings before flipping.

**Tradeoff:** refined version costs ~3 weeks of calendar; shippable-now version costs ~1 PR but every dirty repo's CI breaks Monday morning.

**Recommendation:** refined version. The "Strict over flexible" principle (constitutional principle 2) earns its strictness incrementally rather than by breaking everyone at once.

## P5 — Dashboard alert wiring (deferred per D-17)

30 days after the dashboards are live, review:

- Which dashboards have we actually opened during the 30 days?
- Which panels would have caught a real incident if they'd been alerts?
- What's the right venue — PagerDuty for paging, Slack for visibility, Linear for tracking?

**Shippable now:** pick PagerDuty for the top 3 panels (HTTP error rate, P95 latency, cache hit ratio) — these are universally useful and we know we want pages on them.

**Architecturally refined:** anomaly-band alerts (mean ± 3σ over a rolling 7-day baseline) instead of fixed thresholds. Reduces false positives on legitimate traffic shifts (Black Friday, promo campaigns). Requires a stable historical baseline — needs the 30-day soak before it makes sense.

**Evolution path:** ship fixed-threshold PagerDuty pages for the top 3 panels first. Migrate to anomaly bands once we have 60 days of baseline.

## P6 — Phase 7 OTel Collector swap (gated per D-15)

Documented target state in `stats-lake/ingestion/otel-ingest/COLLECTOR_TARGET.md`. Revisit when **one of**:

- `deco-otel-ingest` Worker hits the cost ceiling (currently far under).
- We need processors the Worker doesn't have (filter by attribute, tail-based sampling, multi-tenant routing).
- We need to ingest from non-CF sources (on-prem services, mobile RUM SDK).

**Shippable now:** keep the Worker. Zero work. The Worker has been deployed, tested, and is producing correct data.

**Architecturally refined:** swap to canonical OTel Collector behind the same OTLP endpoint shape and same ClickHouse schema. Migration path is reversible — A/B test the Collector behind a fraction of traffic before flipping cutover.

**Tradeoff:** the Worker is doing the job. The Collector is doing-the-job-with-someone-else's-roadmap-of-features. Pulling Collector in now is over-engineering until one of the triggers above fires.

## P7 — RUM (Real User Monitoring) — separate scope per user decision
